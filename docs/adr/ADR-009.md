# ADR-009 — recharts for admin data visualization

**Status:** Accepted  
**Date:** 2026-06-26

---

## Context

The admin overview page (`/admin`) renders only plain number cards. To make it a
useful operations dashboard it needs trend visualizations (activity over time,
status breakdowns). No charting library was previously in the dependency set —
only `lucide-react` for icons.

Adding a client-side charting library is a new cross-cutting pattern, so per the
project rule ("a new feature may not introduce a new architectural pattern without
a new ADR") it requires a decision record.

---

## Decision 1 — Adopt recharts, scoped to admin-only client components

`recharts` is added as a dependency and used **only** inside admin-area client
components (`src/components/admin/`). It must not be imported by any public-facing
route or any Server Component.

**Rationale:**
- recharts is a mature, widely-used, dependency-light (D3-submodule based) React
  charting library with an accessible, declarative component API.
- Scoping it to the admin area means the public bundle — which the Lighthouse CI
  budget guards — never pays for it. The chart ships in a code-split chunk that
  only authenticated admins download.

**Trade-offs accepted:**
- ~Adds weight to the admin chunk. Acceptable: the admin area is authenticated,
  low-traffic, and not part of the public performance budget.

---

## Decision 2 — Charts are fed by RSC props, never client-side fetch

Chart components are presentational. The dashboard Server Component loads all data
via repository functions (RSC → repository is allowed) and passes it down as props.
Chart components receive plain arrays and render them; they never call tRPC, `fetch`,
or a repository directly.

**Rationale:**
- Preserves ADR-001 (tRPC is the only client data path) — a chart that renders props
  is not a data path at all.
- Keeps the data-loading story identical to the rest of the admin pages, which all
  load server-side in the RSC and hydrate client widgets with props.

---

## Decision 3 — Every chart ships an accessible non-visual fallback

Any recharts visualization must be accompanied by a screen-reader-accessible
representation of the same data (a visually-hidden `<table>`, or an equivalent
`aria-label` summary). The SVG itself is marked `aria-hidden`.

**Rationale:**
- recharts renders an SVG that is not meaningfully navigable by assistive tech.
- The Lighthouse a11y gate (≥95) and the project's accessibility skill require the
  information to be available without sight. The hidden table carries the real data;
  the chart is decorative on top of it.

---

## Consequences

- `recharts` is a new runtime dependency; `npm audit` must stay clean on it.
- dependency-cruiser should continue to pass: recharts is imported only in
  `src/components/admin/`, never in `src/server/`, `src/lib/`, or public routes.
- Chart components are `"use client"` and guard against the recharts
  `ResponsiveContainer` SSR width warning with a mount flag.
- Future admin charts follow the same three rules: admin-only, props-fed,
  accessible fallback.
