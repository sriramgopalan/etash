# ADR-008 — Widget embed model: iframe + HMAC-HS256 auto-login

**Status:** Accepted  
**Date:** 2026-06-24

---

## Context

Gap 7 adds an embeddable feedback widget: a `<script>` tag that host apps include to
inject a floating button. Clicking it shows a panel that lets users submit feedback
without leaving the host app. Host apps must be able to pre-authenticate their users
so the widget shows their identity rather than a guest prompt.

Two structural questions needed a decision before spec work could start.

---

## Decision 1 — iframe over shadow DOM

The widget panel is rendered inside an `<iframe>` pointing to a dedicated
`/embed/[boardId]` route on the OpenCan server.

**Rationale:**
- The iframe is a separate browsing context. Host-page JavaScript cannot read the
  widget's DOM, cookies, or auth tokens — not even with `document.querySelector`.
- No custom React bundle needs to be shipped or maintained separately; the existing
  Next.js app handles the embed route with a minimal layout.
- iframe postMessage is the well-understood, standardised channel for cross-frame
  communication (close button, height, navigation events).

**Trade-offs accepted:**
- The host page must add `frame-src <opencan-origin>` to its Content-Security-Policy.
  This is documented in the integration guide. It is an opt-in cost, not a security
  regression — a host that has no CSP at all is unaffected.
- The iframe is a separate navigation context, so host-page routing does not reflect
  widget navigation. Acceptable for a feedback widget.

**Rejected: shadow DOM**
A shadow-root approach would run the widget's JavaScript in the host page's execution
context. A compromised or malicious host could inspect auth tokens, override fetch, or
extract session data. For an auth-aware widget, this is an unacceptable risk.

---

## Decision 2 — HMAC HS256 shared secret for JWT auto-login

A single environment variable (`WIDGET_JWT_SECRET`) holds a shared secret used to
verify host-signed JWTs using the HS256 algorithm.

**JWT contract:**

```jsonc
{
  "sub":   "host-user-id",   // host app's user identifier (string)
  "email": "alice@corp.com", // used to create/match an OpenCan User
  "name":  "Alice",          // display name
  "iat":   1719187200,
  "exp":   1719187500        // max 300 seconds (5 minutes) after iat
}
```

`sub` and `email` are required. `name` is optional.

**Token flow:**
1. Host backend signs the JWT with `WIDGET_JWT_SECRET`.
2. Host renders the token into the page (e.g. as a `data-token` attribute).
3. Widget script passes the token as `?token=<jwt>` on the embed iframe URL.
4. OpenCan's `/api/embed-auth` endpoint validates the JWT, finds or creates the
   User by email, creates a NextAuth session, and issues a 302 redirect to the embed
   URL without the token in the URL (so the token does not persist in browser history,
   referrer headers, or server logs).

**Rationale for HS256:**
- One secret to configure; zero key-pair management.
- Sufficient for a single-tenant deployment where the host and OpenCan share a trust
  boundary under the same operator.
- 5-minute expiry limits the window if a token appears in a log or referrer header.

**Known limitation:** HS256 is a symmetric scheme — anyone with the secret can forge
tokens. Suitable for operator-controlled integrations, not for multi-tenant SaaS where
tenants should not hold signing capability over each other's data.

**v2 upgrade path:** Replace `WIDGET_JWT_SECRET` with per-tenant RS256 public keys
stored in the database. The `/api/embed-auth` handler looks up the public key by the
`iss` (issuer) claim. The rest of the flow is unchanged.

---

## Decision 3 — Embed pages opt in to cross-origin framing via CSP

By default, Next.js sets `X-Frame-Options: SAMEORIGIN`, which blocks cross-origin
iframes. The embed routes override this:

```
Content-Security-Policy: frame-ancestors <WIDGET_ALLOWED_ORIGINS>
```

`WIDGET_ALLOWED_ORIGINS` is a space-separated list of host origins the operator
configures (e.g. `https://app.acme.com https://staging.acme.com`). If unset, the
header defaults to `frame-ancestors 'none'` — embedding is disabled entirely.

The `X-Frame-Options` header is omitted on embed routes because modern browsers
use `frame-ancestors` when both are present, and `X-Frame-Options` does not support
multiple origins.

---

## Consequences

- `WIDGET_JWT_SECRET` and `WIDGET_ALLOWED_ORIGINS` are new optional env vars.
  If `WIDGET_JWT_SECRET` is absent, token params are silently ignored (embed works
  as an anonymous guest).
- Host apps must add OpenCan to their `frame-src` CSP directive.
- Users created via JWT auto-login are matched by email. If a host user changes their
  email, they get a new OpenCan user record. Acceptable in v1.
- The embed session is a standard NextAuth session. If the same user is already
  signed in on the main app (same browser, same cookie), the embed inherits that
  session and the JWT auto-login is a no-op (the existing session takes precedence).
